# 《开源技术与应用》课程实验报告

## 开源目的及要求

### 1. 项目开源目的

本项目的开源目的主要包括以下几个方面：

**(1) 降低短链接技术门槛**
短链接服务作为互联网基础设施被广泛使用（微博、短信营销、二维码等场景），但其核心技术——短码生成算法、高并发跳转优化、防缓存穿透策略等——往往封装在商业产品的黑盒中。本项目以 MIT 协议开源全部代码，供开发者和学习者自由查阅、使用、修改。

**(2) 展示 Go 语言在网络中间件领域的工程实践**
Go 语言凭借 goroutine 轻量级并发模型和标准库 `net/http` 对底层协议的掌控力，已成为云原生基础设施的主流语言。本项目以短链接服务为载体，完整展示 Go 在 HTTP 中间件、数据库连接池、异步 IO 管道、Redis Lua 原子操作等场景的工程实现。

**(3) 践行"本地优先"的开源理念**
项目设计原则为 `docker compose up` 一行命令全栈启动，无需云服务器、无需外部 API 依赖（GeoIP 离线解析），确保任何开发者克隆后即可在本地运行完整系统。

**(4) 提供课设级别的教学参考**
项目在代码注释、架构分层、Git 提交历史（7 个独立 Phase 提交）上保持清晰可追溯，为《开源技术与应用》课程提供可参考的工程范例。

## 实验内容

本实验从零构建一个完整的开源短链接服务，涵盖以下技术内容：

1. **后端开发**：Go 语言实现 HTTP Server，包含路由分发、中间件链、数据库连接池、缓存层
2. **短码引擎**：`crypto/rand` 随机数 → Base62 编码 → 唯一性校验，支持自定义短码和保留字过滤
3. **三层缓存架构**：布隆过滤器（防穿透）→ Redis 热数据缓存（防击穿）→ PostgreSQL 持久层
4. **异步数据管道**：Go channel + 批量 worker 实现点击日志的非阻塞写入
5. **限流系统**：Redis Lua 令牌桶 + 内存降级，支持全局/IP/API Key 多维度
6. **认证鉴权**：JWT（golang-jwt）+ bcrypt 密码哈希 + API Key（SHA-256）
7. **User-Agent 解析**：自定义解析引擎，识别设备类型（Desktop/Mobile/Tablet/Bot）、浏览器（Chrome/Safari/Firefox/Edge/WeChat 等 12 种）、操作系统（Windows/macOS/iOS/Android/Linux 等）
8. **GeoIP 离线解析**：基于 IP 段匹配，区分私有网络/云计算/中国 IP，无需外部 API 依赖
9. **智能分发规则引擎**：条件匹配（country/city/os/browser/device/language/referer），支持 eq/neq/contains/prefix 操作符
10. **前端 SPA**：Vue 3 Composition API + Element Plus + ECharts 图表
11. **Docker 容器化**：多阶段构建 Dockerfile + Docker Compose 多服务编排
12. **Git 版本管理**：7 个 Phase 独立提交，保留完整开发轨迹

## 实验过程

### 一、思考过程 & 原理

#### 1.1 为什么选择短链接服务？

短链接看似简单，本质上是一个 **"高并发读、低频写"** 的系统——这与互联网 90% 的服务属于同一类问题。从它可以延伸出：

- **短码生成算法**：哈希冲突处理、分布式 ID 生成（雪花算法）
- **缓存策略**：缓存穿透（布隆过滤器）、缓存击穿（互斥锁）、缓存雪崩（随机 TTL）
- **HTTP 协议**：301 vs 302 语义差异、Keep-Alive 连接复用
- **异步 IO**：Go channel + goroutine 实现非阻塞日志写入
- **限流算法**：令牌桶、滑动窗口、Leaky Bucket 的适用场景对比

选择它作为课设题目，能用较小的代码实现承载较大的技术深度。

#### 1.2 短码生成原理

```
crypto/rand → 随机索引 (0-61) → Base62 字符集查表 → 7 位短码 → 冲突检测 → 重试（最多 10 次）
```

- **Base62 字符集**：`0-9（10个） + A-Z（26个） + a-z（26个） = 62 个字符`
- **7 位组合数**：62⁷ ≈ 3.5 × 10¹²（3.5 万亿）
- **冲突概率**：根据生日悖论，当已有 10 万条短码时，随机碰撞概率 ≈ (10⁵)² / (2 × 3.5 × 10¹²) ≈ 0.14%
- **安全增强**：使用 `crypto/rand`（操作系统熵源）而非 `math/rand`（伪随机）

#### 1.3 三层查询架构原理

```
客户端请求 GET /:code
        │
        ▼
┌──────────────────┐
│ Layer 0: 布隆过滤器 │  判「一定不存在」→ 直接 404，防止无效请求穿透到底层
│ (Redis Bitmap)   │  判「可能存在」→ 进入 Layer 1
└──────┬───────────┘
       │ 可能存在
       ▼
┌──────────────────┐
│ Layer 1: Redis    │  命中 → 直接 302 跳转（< 1ms）
│ (String KV 缓存)  │  未命中 → 进入 Layer 2
└──────┬───────────┘
       │ miss
       ▼
┌──────────────────┐
│ Layer 2: PG       │  查询 → 命中 → 异步回写 Redis（cache-aside）
│ (持久化层)         │        → 302 跳转
└──────────────────┘
```

**布隆过滤器**：使用 k 个哈希函数将短码映射到 m 位 Bitmap。查询时检查所有 k 位——任意一位为 0 则「一定不存在」。本项目设置 m = 1440 万位（针对 100 万元素，误判率 0.1%）。

#### 1.4 点击日志异步管道原理

```
HTTP Handler → chan <- ClickLog (非阻塞 select/default)
                     │
                     ▼
            buffered channel (容量 10,000)
                     │
                     ▼
            batch worker goroutine:
              - 每 500ms 或 攒够 100 条 → PostgreSQL batch INSERT
              - 单条慢于批量，批量优于逐条
```

使用 Go channel 的好处：
- 跳转请求不因日志写入而阻塞（`select default` 在 channel 满时直接丢弃，优先保证跳转性能）
- batch INSERT 大幅减少数据库连接开销
- Worker 单 goroutine 消费，无锁竞争

#### 1.5 Redis Lua 令牌桶原理

```lua
-- 原子操作，避免分布式竞态
local tokens = redis.call("HGET", key, "tokens")    -- 当前令牌数
local last = redis.call("HGET", key, "last")         -- 上次填充时间
local elapsed = now - last
local refill = math.floor(elapsed * rate / window)   -- 按时间比例补充
tokens = math.min(burst, tokens + refill)
if tokens < 1 then return 0 end                      -- 限流
redis.call("HSET", key, "tokens", tokens - 1)        -- 消耗 1 个令牌
```

使用 Lua 脚本的原因：GET + 计算 + SET 三步操作在 Redis 中如果分开执行会有竞态，Lua 脚本保证原子性（Redis 单线程执行 Lua）。

### 三、程序流程

#### 3.1 系统启动流程

```
main() 启动
    │
    ├─► config.Load()               加载 .env 环境变量
    ├─► repository.Connect()        连接 PostgreSQL
    ├─► repository.RunMigrations()  执行 DDL 建表
    ├─► cache.NewRedisCache()       连接 Redis
    ├─► cache.NewBloomFilter()      创建布隆过滤器
    │       └─► bloom.LoadAll()     加载全量短码到 Bitmap
    ├─► 创建 Service 层实例          组装业务逻辑
    ├─► 启动 ClickLogWorker         异步日志消费 goroutine
    ├─► 注册路由 (chi.Router)
    │       ├─► /health            GET  健康检查
    │       ├─► /api/v1/*          API 路由组
    │       │       ├─► Public     POST /links, /auth/register, /auth/login
    │       │       └─► Protected  GET/PATCH/DELETE /links/*, /dashboard/*
    │       └─► /{shortCode}       GET  短码跳转（兜底路由）
    ├─► http.ListenAndServe()       启动 HTTP 服务
    └─► signal.Notify()             监听 SIGINT/SIGTERM 优雅关闭
```

#### 3.2 创建短链接流程

```
POST /api/v1/links {"url":"https://example.com/..."}
    │
    ▼
handler.Create()
    ├─► json.Decode → model.CreateLinkReq
    ├─► 验证 URL（格式/http/https/非本域）
    ├─► 自定义码？→ 格式校验 + 保留字检查 + 唯一性校验
    ├─► 随机生成？→ service.Generate()
    │       └─► crypto/rand → Base62(7位) → 冲突检测 → 重试(≤10次)
    ├─► 可选字段处理：过期时间/RFC3339解析、密码/bcrypt哈希、规则/JSON序列化
    ├─► repo.Create() → INSERT INTO links RETURNING id
    ├─► cache.SetURL() → Redis 缓存新链接
    └─► 返回 JSON {short_url, short_code, ...}
```

#### 3.3 短码跳转流程（热路径，核心性能瓶颈）

```
GET /A3xK9q
    │
    ▼
handler.ServeRedirect()
    ├─► Bloom.MightExist("A3xK9q")
    │       └─► k 个哈希 → GetBit × k → 任一为 0？
    │               ├─► 是 → 404（一定不存在）
    │               └─► 否 → 继续
    ├─► Redis.GetURL("A3xK9q")
    │       └─► 命中？→ 302 Redirect（< 1ms，热路径）
    │       └─► 未命中？→ 继续
    ├─► repo.GetByShortCode("A3xK9q")
    │       └─► 找到 → 回写 Redis → 继续
    │       └─► nil → 404
    ├─► 过期检查：ExpireAt != nil && time.Now().After(*ExpireAt) → 410 Gone
    ├─► 激活检查：!IsActive → 410 Gone
    ├─► 密码检查：PasswordHash != "" → 展示密码输入页 / 验证密码
    ├─► 规则引擎：ApplyRules() → 匹配规则 → 替换 targetURL
    ├─► 异步记日志：chan <- ClickLog (select default 防阻塞)
    └─► http.Redirect(w, r, targetURL, 302/301)
```

#### 3.4 点击日志异步管道流程

```
Redirect Handler → clickChan <- ClickLog
                        │
    ┌───────────────────┘
    ▼
clickLogWorker (独立 goroutine)
    │
    ├─► select {
    │       case click := <-clickChan:
    │           batch = append(batch, click)
    │           if len(batch) >= 100 → flushBatch()
    │       case <-ticker.C (500ms):
    │           if len(batch) > 0 → flushBatch()
    │   }
    │
    ▼
flushBatch()
    └─► repo.BatchInsert(ctx, batch)
        └─► pgx.Batch → SendBatch → 批量 INSERT
            （100 条一次提交，减少 DB 交互次数）
```

### 四、应用场景

#### 4.1 营销归因分析

企业的推广链接分散在微博、公众号、抖音、邮件等多个渠道。为每个渠道生成不同短码：

```
https://s.id/A3x → 微博推广
https://s.id/B7y → 公众号文章
https://s.id/C2z → KOL 视频简介
```

通过 Dashboard 查看各渠道点击量、地理分布、设备分布，判断 ROI 最高的投放渠道。

#### 4.2 智能应用分发（Deep Link）

同一个短链接，不同设备跳转到不同应用商店：

```
https://s.id/app
    ├─ CN + iOS     → App Store
    ├─ CN + Android → 应用宝/华为应用市场
    ├─ US + iOS     → US App Store
    └─ 其他          → 官网 H5 下载页
```

通过规则引擎的条件匹配实现（已在项目中实现）。

#### 4.3 社交媒体分享

微博 140 字限制、短信字数计费——长 URL 占用大量字符。短链接压缩至 ~20 字符，节省空间。配合二维码，线下场景扫码跳转。

#### 4.4 敏感链接安全预览

短链接被滥用于钓鱼攻击。本项目支持：
- 密码保护：访问时需输入密码
- 预览页：展示目标 URL 全貌，用户确认后再跳转
- 域名黑名单：自动拦截已知恶意域名
- 过期时间：临时链接定时失效

#### 4.5 开放 API 平台

将短链接能力封装为 RESTful API，提供 API Key 管理、调用量统计、限流配额——第三方开发者可集成到自己的应用中。

#### 4.6 企业级部署场景

- **内网短链服务**：企业内网部署，域名如 `go.company.com`，减少长 URL 在邮件/文档中的视觉干扰
- **合规审计**：所有点击记录可追溯，满足安全审计要求
- **自定义域名**：绑定企业自有域名，保持品牌一致性
